The Problem

Single AI calls don't scale to real work

Complex knowledge work requires depth across multiple domains simultaneously — research, analysis, code, synthesis. A single LLM prompt cannot do all of it well.

  • Context windows overflow on large tasks
  • Monolithic prompts produce shallow, unfocused output
  • No parallelism — every step is sequential
  • No accountability — opaque reasoning, no audit trail
  • Cost blindness — no per-task budget control
  • Human oversight is absent until the very end
The insight: The best teams assign specialized people to parallel workstreams with a coordinator. AI agents should work the same way.

❌ Monolithic Approach

One prompt, one model, one response. Hits context limits. Sequential. No specialization. No transparency.

✅ Multi-Agent Approach

Specialized agents run in parallel. Each scoped to its domain. Coordinator synthesizes results. Streaming progress. Human checkpoints.

Overview

What is AgentFlow?

AgentFlow is an open-source orchestration engine that decomposes a natural-language task into a directed acyclic graph of specialized agent subtasks, executes them in parallel, and streams every state transition to the client in real time.

🧩

Composable

New agents added via a single YAML manifest file. No orchestrator code changes required.

⚡

Parallel

DAG-based scheduler dispatches all independent subtasks simultaneously, cutting wall-clock time dramatically.

👁️

Transparent

Every state transition — plan creation, agent progress, cost, errors — streams live to the client via SSE.

🛡️

Budget-Safe

Per-subtask USD budgets enforced in real time. Human approval gate before budget expansion.

🔧

Tool-Rich

Built-in tools (bash, file I/O, web search) plus MCP server integration for external capabilities.

🔄

Resilient

Automatic retry, fallback agent routing, and continuation for partial results near iteration limits.

Business Value

Why it matters for the organization

Accelerate knowledge work

Run research, coding, analysis, and writing in parallel instead of sequentially. A task that takes 30 minutes of serial LLM calls completes in 5 minutes with parallel agents.

Predictable AI costs

Every run has a declared USD budget cap. Prompt caching cuts token costs up to 90% on repeated system prompts. Model routing assigns cheap models to simple subtasks.

Human oversight built in

The Human-in-the-Loop pattern pauses execution for human approval before sensitive or budget-exceeding actions — compliance-friendly by design.

Extensible platform

Add a domain expert agent in one YAML file. No changes to the orchestrator, no redeployment of existing agents. Ship new capabilities in hours.

Target use cases

  • Automated research & competitive analysis
  • Multi-file codebase generation and review
  • Data pipeline creation with validation
  • Technical documentation with factual grounding
  • Business analysis with web + internal data sources

Integration model

Standard REST + SSE API. Submit a task via POST /runs, stream results via GET /runs/{id}/stream. Integrates with any frontend or backend in minutes.

Design Principles

Six load-bearing design decisions

📋

1. Declarative agents

Every agent is a YAML manifest. Capabilities, tools, system prompt, iteration limits — all config, not code. Add agents without touching the engine.

🔒

2. Stateless agents

Agents receive all needed context via a typed TaskEnvelope. No shared mutable state between agents. No direct inter-agent calls.

🧠

3. LLM-powered planning

The decomposition step is itself an LLM call. The planner reads the agent registry and workspace, then produces a JSON execution plan with dependency declarations.

📡

4. Streaming-first

Every state transition emits a typed SSE event. The client is never waiting blind — it sees the plan, each dispatch, each completion, and costs in real time.

💰

5. Budget awareness

Each subtask gets a fractional budget slice. Real-time USD tracking. Budget exhaustion triggers a human approval gate before execution continues.

🔁

6. Swappable internals

TaskBus, StreamEmitter, RunContext all have clean interfaces. In-process asyncio today; Redis Streams on day one of scale-out.

Declarative Agents

Agent manifest — behavior as configuration

Each agent is fully declared in a YAML file. The orchestrator loads all manifests at startup — no code changes needed to add or modify an agent.

agent_id: ResearchAgent
domain: Information Retrieval
capabilities:
  - web_research
  - document_synthesis
  - citation

tools:
  - web_search
  - fetch_url
  - file_read
  - file_write

mcp_servers:
  - name: knowledge-base
    transport: stdio

skills:
  - research-methodology

max_iterations: 8
fallback_for: KBAgent

system_prompt: |
  You are an expert research analyst...

What a manifest controls

Capability index

The registry builds a capability → [agent_ids] map. The LLM planner reads this roster to decide which agent handles which subtask.

Tool scope

Agents only receive tools listed in their manifest. The tool registry filters globally registered tools — enforced isolation.

MCP integration

Agents can declare external MCP servers (stdio or SSE). The engine connects at runtime and injects their tools into the agentic loop automatically.

Fallback routing

fallback_for field means this agent is the retry target if the primary agent fails — declared, not hardcoded.

Result: A domain expert can ship a new AI capability by writing a 30-line YAML file. No Python required.

Retry logic

Configurable task_max_retries with exponential backoff. On final failure, registry finds a fallback_for agent.

Planning

LLM-powered task decomposition

The planner is itself an agentic loop — it receives the task, the agent registry roster, and a workspace exploration toolkit, then produces a structured execution plan.

Phase 1 — Exploration (ReAct loop)

Planner calls file_read, bash_exec, web_search to understand the workspace before committing to a plan (max 5–8 tool calls).

Phase 2 — Decomposition

LLM outputs a JSON plan. Rules enforced by the system prompt: budget fractions must sum to 1.0, prefer parallel over sequential, always add verification subtasks after file generation.

Phase 3 — Expansion

Agents with a decomposition_prompt (e.g. CodeAgent) can further split their assigned subtask into micro-tasks before execution begins.

Scheduler

DAG-based dependency scheduling

NetworkX DiGraph

The scheduler builds a directed acyclic graph from the subtask dependsOn declarations. Cycle detection runs at plan ingestion — a circular dependency rejects the plan.

Ready computation

On each loop iteration: ready = graph.ready(completed, failed). All ready subtasks dispatched simultaneously via asyncio.gather().

Context inheritance

When a subtask has exactly one dependency, it inherits the full message thread — no redundant re-reading of prior output. Multi-dependency subtasks receive a synthesized text summary.

Wall-clock impact: A 3-agent plan where two are independent runs in the time of the longest single agent — not their sum.

Agentic Loop

The agent execution loop

async def _agentic_loop(self, envelope):
  messages = build_initial_messages(envelope)

  for iteration in range(self.max_iterations):
    response = await self.llm.call(
      model=self.model,
      system=self.system_prompt,
      messages=messages,
      tools=self.tools,          # scoped to manifest
      betas=["prompt-caching-2024-07-31"]
    )

    # track cost in real time
    self._accumulate_tokens(response.usage)

    if response.stop_reason == "end_turn":
      return AgentResult(status="success", ...)

    if response.stop_reason == "tool_use":
      tool_results = []
      for block in response.content:
        result = await self._call_tool(block)
        tool_results.append(result)
      messages.extend([response, tool_results])

    if self._budget_exhausted():
      return AgentResult(status="partial", ...)

  return AgentResult(status="partial", ...)

Standard ReAct pattern

Think → Act (tool call) → Observe (tool result) → repeat. The loop continues until end_turn or a stop condition.

Three stop conditions

  • end_turn — model finished → success
  • Budget exhausted → partial result
  • Max iterations hit → partial result

Tool result capping

Tool outputs are truncated to 8K chars before injection into the message thread — prevents context bloat on large file reads or web pages.

Continuation support

A partial result carries its full message history. The engine can re-dispatch with "Continue where you left off..." — up to 3× per subtask.

Streaming

Real-time streaming with Server-Sent Events

Every state transition in the pipeline emits a typed SSE event. The client never waits blind — it receives incremental progress from plan creation through final report.

SSE endpoint

@router.get("/runs/{run_id}/stream")
async def stream_run(run_id: str):
  emitter = registry.get_emitter(run_id)
  return EventSourceResponse(
    emitter.event_generator()
  )

Typed event sequence

→ run:started {run_id, task, budget_usd}
→ plan:created {subtasks: [...]}
→ task:dispatched {agent_id, subtask_id}
→ agent:progress {message, cost_usd}
→ task:complete {output, tokens}
→ task:dispatched {agent_id: Writer}
→ task:complete {output}
→ run:complete {report, total_cost}

StreamEmitter internals

Each run gets its own asyncio.Queue. The event_generator() async-yields from the queue. A sentinel value signals stream end.

Events are also persisted to a JSONL file — the client can replay the full event log at any time via GET /runs/{id}/events.

Per-event structure

{
  "type": "task:complete",
  "seq": 7,
  "agent_id": "ResearchAgent",
  "subtask_id": "st_1",
  "timestamp": "2026-06-29T...",
  "data": { ... }
}
Design note: The StreamEmitter and StreamRegistry use a clean interface — swapping the asyncio.Queue for a Redis Stream unlocks multi-replica deployments without changing any consumer code.
Prompt Caching

Prompt caching — the biggest cost lever

System prompts and skills are often hundreds of tokens — repeated on every agentic loop iteration. Anthropic's prompt caching reduces those repeated tokens to cache reads at 10× lower cost.

How it works

Mark the static parts of the prompt with cache_control: {type: "ephemeral"}. Claude caches the KV activations server-side. Subsequent calls within 5 minutes reuse the cache.

What gets cached in AgentFlow

  • Agent system prompt (often 300–600 tokens)
  • Skill preamble text (reusable knowledge injected at runtime)
  • Tool definitions (static schema blocks)
  • Prior message history on continuation runs

Cost comparison

$3.00
per 1M input tokens
$0.30
per 1M cache read tokens

Implementation in LLM client

# System block with cache breakpoint
system = [
  {
    "type": "text",
    "text": agent.system_prompt,
    "cache_control": {"type": "ephemeral"}
  }
]

# Tool definitions with cache breakpoint
tools = [
  {
    ...tool_schema,
    "cache_control": {"type": "ephemeral"}
  }
  for tool in agent.tools
]

# Token tracking from response
cache_creation = usage.cache_creation_input_tokens
cache_read     = usage.cache_read_input_tokens
cost = (
  cache_creation * 3.75
  + cache_read   * 0.30  # 10× cheaper
  + output_toks  * 15.0
) / 1_000_000
An agent with a 500-token system prompt running 8 iterations: without caching = 4,000 input tokens; with caching = 500 write + 3,500 reads = ~87% cost reduction on the system prompt portion.
Human-in-the-Loop

Human-in-the-Loop — safety by design

The HITL pattern pauses the orchestrator and asks a human before budget-exceeding or sensitive actions proceed. The run state is preserved across the pause.

Current implementation — Budget gate

When a subtask exhausts its allocated USD budget slice, the orchestrator emits a human_input_request SSE event, acquires an async lock, and waits (up to 30 min) for a response via POST /runs/{id}/input.

Only one budget question surfaces at a time — the lock ensures serialization even when multiple subtasks exhaust simultaneously.

The interaction flow

SSE → human_input_request {budget_used, budget_cap}
↳ client shows approval dialog
POST /runs/{id}/input {"approved": true, "amount": 2.00}
↳ engine resumes with new budget
SSE → task:dispatched (continuation)

Roadmap: General interrupt mechanism

The gap analysis identifies expanding HITL beyond budget to any irreversible action (bash_exec, file_write). A general InterruptRequest model would carry:

  • Trigger reason (budget / irreversible action / low confidence)
  • Proposed action + full tool input
  • Reasoning trace and cost so far
  • Options: approve / reject / modify / restart
  • SLA timeout with safe default
Research finding (Wang & Wang 2026): Agents given only output-level human feedback achieved 0% task completion due to "failure mode oscillation." The interrupt payload must include reasoning trace — humans can't fix what they can't see.

Compliance angle

Every HITL event is logged to the run's event JSONL — a complete audit trail of what the human approved, when, and at what cost point.

Skills

Skills — reusable knowledge libraries

Skills are shared prompt fragments — domain knowledge, coding standards, methodology guides — that any agent can load at runtime. They are injected into the system prompt before the agentic loop starts.

How skills are loaded

If a manifest declares skills: [python-coding], the engine injects a read_skill tool and a system prompt instruction to load it before starting work. The agent reads the skill file on its first tool call.

Example skill: python-coding

Injects project-specific standards: type hints, black formatting, test patterns, import conventions. All CodeAgent subtasks share this knowledge automatically — without repeating it in every agent system prompt.

Caching synergy

Because skills are stable text, they're ideal cache candidates. Once the skill text is in the context window, subsequent loop iterations read it from the prompt cache — near-zero cost.

Example skill file — skills/python-coding.md

# Python Coding Standards

## Type hints
Always annotate function signatures.
Use `Optional[T]` not `T | None` for compatibility.

## Testing
Write pytest tests alongside implementation.
Use `pytest.mark.asyncio` for async functions.
Mock external services with `unittest.mock`.

## Formatting
black line length: 88
isort profile: black

## Error handling
Raise domain-specific exceptions.
Never swallow exceptions silently.
Log with structlog, not print().
Practical benefit: A team's engineering standards live in one file. Every CodeAgent subtask — in any run, by any user — automatically follows them. Update the skill file; every future run gets the update.
MCP Integration

MCP — connecting to the tool ecosystem

The Model Context Protocol (MCP) is an open standard for exposing tools to LLMs. AgentFlow agents can connect to any MCP server — stdio subprocess or SSE endpoint — declared in their manifest.

Runtime connection

When an agent starts, the engine opens MCP connections for each server in the manifest's mcp_servers list. Tool schemas are fetched and merged with built-in tools before the agentic loop starts.

Supported transports

stdio subprocess SSE endpoint

Stdio: spawns the MCP server as a child process. SSE: connects to a running HTTP MCP server. Both present identical tool interfaces to the agent.

Example MCP servers in use

  • Knowledge base search (KB agent)
  • Internal database query tools
  • Proprietary API wrappers
  • Slack / GitHub / Jira connectors

Manifest declaration

mcp_servers:
  - name: knowledge-base
    transport: stdio
    command: python
    args: ["-m", "kb_server"]

  - name: github-tools
    transport: sse
    url: http://mcp-github:8080/sse

Engine connection logic

for server_cfg in manifest.mcp_servers:
  client = await MCPClient.connect(server_cfg)
  mcp_tools = await client.list_tools()
  all_tools.extend(mcp_tools)
# mcp_tools merged with built-ins
# agent sees unified tool list
Open standard: Any tool server implementing MCP — from the community or built in-house — integrates with zero framework changes.
Context

Context propagation between agents

Agents are stateless — but downstream agents need upstream results. RunContext stores each completed subtask result and provides two propagation modes depending on the dependency graph shape.

Mode A — Text summary (multi-dependency)

build_prior_results(dep_ids) concatenates the text output from all upstream subtasks into a structured JSON summary and injects it as user context. Efficient for fan-in patterns.

Mode B — Full message thread (single dependency)

build_prior_messages(dep_id) passes the entire conversation thread from the upstream agent — tool calls, tool results, and all. The downstream agent sees the reasoning, not just the conclusion.

User context threading

The initial user_context dict (submitted with the run) propagates to every agent via the task envelope. Global facts available to all subtasks without re-fetching.

RunContext key methods

class RunContext:
  # Store result after subtask completes
  def store_result(self, subtask_id, result):
    self._results[subtask_id] = result
    self._persist_to_jsonl(result)  # disk

  # Multi-dependency: synthesised summary
  def build_prior_results(self, dep_ids):
    parts = []
    for dep_id in dep_ids:
      r = self._results[dep_id]
      parts.append({"id": dep_id,
                    "output": r.output["text"]})
    return json.dumps(parts)

  # Single dependency: full message thread
  def build_prior_messages(self, dep_id):
    r = self._results[dep_id]
    return r.messages  # full history

  # Budget tracking
  def remaining_budget_usd(self):
    spent = sum(r.cost_usd for r
                in self._results.values())
    return self.budget_usd - spent
Cost Strategy

Model routing — right-sizing compute to the task

Not every subtask needs the same model. A formatting pass is not the same as multi-step code generation. AgentFlow supports per-manifest model overrides, with complexity-based routing on the roadmap.

Current — per-manifest override

Manifest declares model: claude-haiku-4-5 to override the global default. The KB search agent, reporter, and simple extraction agents run on Haiku; CodeAgent runs on Sonnet.

Roadmap — complexity routing

The planner will score each subtask's complexity tier during decomposition. The dispatcher selects the model from a config map — no per-agent manifest change needed.

Deployment — Scenario 1

Single-server deployment

When to use

  • Development and experimentation
  • Internal team tooling (low concurrency)
  • Proof-of-concept demos
  • <20 concurrent runs expected

Stack

  • Python 3.12 + uv
  • FastAPI + uvicorn (single worker)
  • In-process asyncio concurrency
  • Local disk for run persistence
  • ANTHROPIC_API_KEY in .env

Limitations

  • State lost on server restart
  • Cannot scale horizontally
  • No auth or tenant isolation
  • Unbounded asyncio task accumulation under load

Deployment — Scenario 2

Horizontally scalable deployment

Key changes from single-server

  • RunContext → Redis hash per run
  • StreamEmitter → Redis Stream (XADD/XREAD BLOCK)
  • TaskBus → Redis pub/sub or LPUSH/BRPOPLPUSH
  • HITL async events → Redis pub/sub channel
  • Concurrency semaphore → HTTP 429 on overload

Why this unlocks scale

Any replica can handle any request — no sticky sessions. A run created on Replica 1 can stream to Replica 2. A restart of any replica loses zero in-flight state. Run behind any standard load balancer.

Auth layer (required for multi-tenant)

API key middleware. Redis key prefix per tenant: {tenant}:run:{id}:*. Postgres RLS enforces row-level tenant isolation on history tables.

Deployment — Scenario 3

Cloud-native / Kubernetes deployment

For high-volume or long-horizon runs, decouple submission from execution using a task queue. Kubernetes provides pod autoscaling on queue depth.

Architecture additions

  • Celery workers — replace BackgroundTasks; pull from Redis/RabbitMQ
  • API pods — only handle HTTP + SSE; no LLM calls
  • Worker pods — only handle LLM calls + tool execution
  • HPA — autoscale workers on Celery queue depth metric
  • Celery Beat — scheduled or recurring runs

Why Celery for this pattern

Celery's chord and group primitives map directly onto the DAG fan-out pattern. A chord fans subtasks out to workers and calls a callback when all complete — the orchestrator fan-in step.

Observability stack

  • OpenTelemetry → Jaeger (distributed traces)
  • Prometheus + Grafana (LLM cost metrics)
  • LangSmith / Langfuse (LLM trace capture)

Summary

Key takeaways

1. Declarative agents are the right abstraction

YAML manifests make agents first-class configuration objects. Domain experts can add AI capabilities without touching orchestration code.

2. LLM planning + DAG scheduling = natural parallelism

The planner naturally discovers independent subtasks; the DAG scheduler runs them simultaneously. Wall-clock time drops dramatically without any special effort.

3. Streaming + prompt caching = great UX at low cost

Users see live progress. Prompt caching cuts token costs up to 90% on repeated system prompts. These are independent benefits that compound.

4. Human-in-the-Loop is a feature, not an afterthought

Budget approval gates + rich interrupt payloads make AI agents safe for irreversible actions. Compliance teams can audit every human decision.

5. Clean interfaces enable incremental scaling

RunContext, StreamEmitter, TaskBus have swappable implementations. Swap asyncio for Redis; nothing else changes. Prototype to production is a configuration change.

6. The foundation is solid — gaps are well-understood

The gap analysis maps exactly what needs to change for each deployment scenario. Two sprint cycles get AgentFlow from prototype to production-grade.

The single sentence

AgentFlow is a composable, streaming, budget-aware multi-agent orchestration platform built on solid architectural instincts — a prototype that is two focused sprints away from running safely in production at scale.
